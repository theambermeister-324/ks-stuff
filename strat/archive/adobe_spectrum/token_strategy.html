1. Current State Analysis

Adobe's Token Architecture

Source of Truth: Adobe/spectrum-design-data (public GitHub mono-repo)

  • Contains spectrum-tokens npm package (JSON format)
  • Component options via JSON schema
  • Custom JSON format using Style Dictionary with nested global properties
  • Multi-dimensional values (theme, density, platform) per token

Platform Implementations
Critical Insight

Figma cannot serve as source of truth due to platform limitations. The custom UUID-based sync script bridges Token Studio and the repo, maintaining data structure integrity across incompatible formats.

Strategic Recommendations

Token Integration Approach for Knapsack Implementation
Adobe Spectrum

Recommendation 1: Consume-First Integration

R1 Priority Recommended

Configure Knapsack to consume the published spectrum-tokens npm package rather than attempting bi-directional authoring in R1.

Rationale

  • Respects Adobe's established source of truth
  • Avoids disrupting the existing Token Studio → GitHub workflow
  • Aligns with R-07 mitigation strategy
  • Reduces R1 complexity while delivering immediate value

Implementation Path

  1. Configure npm package consumption from @adobe/spectrum-tokens
  2. Build token documentation views displaying semantic tokens with mode variants
  3. Implement visual token browser with light/dark, desktop/mobile previews
  4. Create token-to-component mapping documentation

Recommendation 2: Multi-Dimensional Display

Challenge: Adobe's tokens encode multiple dimensions (theme × density × platform) as single tokens with multiple values.

Dimension Display Strategy User Interaction
Theme (light/dark) Side-by-side preview panels Toggle switch
Density (desktop/mobile) Contextual view tabs Viewport selector
Combined states Matrix visualization Multi-toggle dashboard
Technical Validation Required

Work with FDE during R0 to validate Knapsack's multi-modal token rendering capabilities before finalizing display approach.

Recommendation 3: Token Wayfinding (R2/R3)

Opportunity: Adobe is building AI-powered token wayfinding via Slack chatbot. Knapsack's MCP Beta Program creates a natural integration point.

Recommended Sequence

Release Capability
R1 Document token usage patterns and designer decision trees manually
R2 Integrate token documentation with contribution workflows
R3 Explore MCP integration for token recommendation queries

Recommendation 4: Defer Authoring

Risk Mitigation for R-07 Post-MVP

Position authoring capability as post-MVP scope. Premature authoring risks sync conflicts, schema violations, and governance gaps.

Prerequisites for Future Authoring

Validated UUID synchronization compatibility
Schema validation integration matching Adobe's enforcement
Custom diffing engine integration for release notes
Governance workflow for token contribution approval
Rollback procedures documented and tested

Clear Boundary Definition

Tool Responsibility
Token Studio Authoring, design iteration, Figma sync
Knapsack Documentation, consumption, wayfinding, governance

Technical Architecture Decision

Token Consumption Method Selection
Adobe Spectrum
Decision Required: R0 Gate

Token Consumption Method Options

Option A: npm Package Consumption Recommended for R1

Consume @adobe/spectrum-tokens published package

Advantages
  • Stable, versioned releases
  • Matches implementation team consumption pattern
  • No additional security review required
  • Documentation matches production reality
Limitations
  • Update delay (requires publish cycle)
  • No preview of upcoming changes
  • Dependent on Adobe's release cadence
Option B: Direct Repo Sync Alternative

Connect Knapsack directly to spectrum-design-data repository

Pros: Real-time updates, pre-publish visibility
Cons: Security review complexity, repo access approval required
Option C: Hybrid Approach R2+ Consideration

npm for stable documentation; repo for preview/staging environments

Pros: Best of both, supports staged rollout
Cons: Higher maintenance, dual pipeline management
