0.5From Design Systems to Product Context

"Design systems aren't obsolete. They're foundational. But you create a foundation to support something larger."
‚Äî Robin Cannon, VP Product, Knapsack

Design systems emerged to solve real problems: fragmented interfaces, repeated implementation work, fractured user experiences with inconsistent behavior. That work still matters‚Äîmore than ever. But the environment has changed.

AI Doesn't Consume Components‚ÄîIt Consumes Context

When we talk about AI using design systems as inputs, we're missing the full picture. What AI actually consumes, and amplifies, is context:

  • Which decisions are encouraged vs. merely permitted
  • Where the system is strict and where flexibility is expected
  • How to handle accessibility tradeoffs
  • Which interaction patterns are preferred‚Äîand why
  • Tone and voice changes across different moments
  • Historical exceptions and the justifications behind them

This context doesn't live cleanly in a component library. It's scattered across documentation, Slack discussions, tribal knowledge, and decisions we forgot to codify. Humans survive ambiguity; machines producing at scale cannot.

The Risk of Accelerated Drift

Without strong product context, AI creates divergence rather than coherence:

  • Prompts are local decisions ‚Äî Each AI interaction makes isolated choices
  • Outputs are reasonable in isolation ‚Äî Generated UI looks correct at first glance
  • Product drift moves from subtle to structural ‚Äî Without governed context, inconsistency compounds

AI-generated UI often feels "superficially correct but wrong" because AI follows visible rules and misses invisible constraints. Design systems aren't about how things look‚Äîthey're about how we propagate our decisions.

Product Context Is the Work

If a design system is the foundation, product context is the structure built upon and around it:

Dimension Design System Product Context
Visual Tokens, components, layouts + Why these values were chosen, when to deviate
Behavioral Interaction patterns + Preferred patterns vs. permitted alternatives, accessibility decisions
Content Typography, spacing + Tone, voice, language constraints, content principles
Governance Contribution guidelines + Review expectations, regulatory boundaries, risk tolerance
History Changelog + Why exceptions exist, precedent for edge cases

The Role Shift

This reframes the roles and responsibilities of design system teams:

  • Maintain intent, not artifacts ‚Äî The "why" matters more than the "what"
  • Define boundaries, not enforce consistency ‚Äî Where is flexibility encouraged?
  • Create machine-readable context ‚Äî As much as human documentation

It's less about expanding control and more about expanding clarity. AI needs better constraints, not more pixels.

PCRI measures this broader scope‚Äînot just whether you have components, but whether your context is operational enough to govern AI outputs at scale.

1.2 Why Five Pillars?

Research on successful product context transformations reveals five distinct capability domains:

  1. Context Foundation ‚Äî The governed context substrate (tokens, components, decisions, constraints, accessibility)
  2. AI Readiness ‚Äî Ability to leverage AI with machine-readable context and governed quality
  3. Governance ‚Äî Organizational authority, decision rights, and executive sponsorship
  4. Delivery ‚Äî Workflow consistency, deployment frequency, and cross-functional collaboration
  5. Culture ‚Äî Making implicit context explicit through contribution models, psychological safety, and information flow

These pillars are:

  • Mutually exclusive: Each measures a distinct capability
  • Collectively exhaustive: Together they cover the full readiness landscape
  • Independently measurable: Each can be assessed with specific questions
  • Interdependent: Progress in one affects achievable outcomes in others

Pillar Weight Rationale
Culture 25% Foundational enabler‚Äîpsychological safety and contribution culture are the strongest predictors of sustainable adoption per Project Aristotle research.
Foundation 20% Technical substrate‚Äînecessary for any product system work. Token architecture, components, and accessibility.

2.2 Weight Rationale

Why Culture Leads (25%)

The 25% weight on Culture reflects empirical research:

  • Google's Project Aristotle identified psychological safety as the #1 predictor of team effectiveness
  • Westrum's research shows information flow patterns predict organizational performance
  • Contribution models determine long-term sustainability of product systems

Foundation, AI Readiness, and Governance are equally weighted (20% each) because:

  • Foundation without governance = technical debt without adoption
  • Governance without foundation = policy without enforcement
  • AI Readiness requires both foundation and governance to deliver value

Delivery is weighted lowest (15%) because:

  • It is most influenced by the other four pillars
  • It tends to improve naturally as F/AI/G/C mature
  • It is partially gated by governance maturity

3Gated Progression Model

3.1 The Problem with Ungated Scoring

A simple weighted average allows organizations to score high on AI Readiness while having no foundation‚Äîthis is misleading because:

  • AI tools without governed context produce inconsistent output
  • High AI adoption with low foundation = higher rework costs
  • The score should reflect achievable value, not just current state

3.2 Gating Rules

PCRI implements two progression gates:

5Composite Score Calculation

5.1 Calculation Steps

  1. Sum weighted raw points per pillar
  2. Calculate max possible weighted points per pillar
  3. Normalize each pillar to 0-100
  4. Apply gating discounts if thresholds not met
  5. Calculate weighted composite (PCRI)

5.2 Formula

PCRI Calculation Formula
PCRI = (Foundationeff √ó 0.20) + (AIeff √ó 0.20) + (Governanceeff √ó 0.20) + (Deliveryeff √ó 0.15) + (Cultureeff √ó 0.25)

Where _eff = effective score after gating.

10Validation & Calibration

10.1 Validation Approach

PCRI validity is assessed through:

  1. Face validity: Expert review of question relevance
  2. Content validity: Coverage of all relevant capability domains
  3. Criterion validity: Correlation with engagement outcomes
  4. Construct validity: Factor analysis of pillar structure
  5. Research validity: Alignment with DORA, Project Aristotle, Westrum findings

10.2 Calibration Process

  1. Run assessment on 10+ known customers with known outcomes
  2. Compare PCRI predictions to actual engagement results
  3. Identify questions with low predictive value
  4. Adjust weights or replace questions
  5. Document changes for version tracking

12Ethical Considerations

12.1 Assessment Bias

PCRI is designed to minimize:

  • Question bias: Neutral language, no leading questions
  • Cultural bias: Applicable across industries and geographies
  • Size bias: Questions work for 100-person and 100,000-person orgs

12.2 Data Privacy

  • No PII required for scoring
  • Organization data aggregated for benchmarks
  • Individual responses not shared externally

12.3 Score Transparency

  • Scoring algorithm is documented
  • Weights are disclosed
  • Gating logic is explained in results

13Technical Implementation

13.1 Scoring Algorithm

// PCRI Scoring Algorithm
function calculatePCRI(answers) {
  // Step 1: Calculate raw pillar scores
  let pillars = { foundation: 0, ai: 0, governance: 0, delivery: 0, culture: 0 };
  let maxes = { foundation: 0, ai: 0, governance: 0, delivery: 0, culture: 0 };
  
  for (question in questions) {
    score = answers[question] * questionWeights[question];
    max = maxScore[question] * questionWeights[question];
    pillars[question.category] += score;
    maxes[question.category] += max;
  }
  
  // Step 2: Normalize to 0-100
  let norms = {};
  for (pillar in pillars) {
    norms[pillar] = (pillars[pillar] / maxes[pillar]) * 100;
  }
  
  // Step 3: Apply gating
  if (norms.foundation < 40) {
    norms.ai = norms.ai * 0.5;
  }
  if (norms.governance < 30) {
    norms.delivery = norms.delivery * 0.7;
  }
  
  // Step 4: Calculate composite
  pcri = (norms.foundation * 0.20) +
         (norms.ai * 0.20) +
         (norms.governance * 0.20) +
         (norms.delivery * 0.15) +
         (norms.culture * 0.25);
  
  return round(pcri);
}
