Contributing to Your Design System

Design systems evolve ‚Äî and so do the ways teams contribute to them. If you're just getting started, our goal is simple: help you move fast and learn what matters. As your system grows and more teams rely on it, the contribution process naturally becomes a little more structured ‚Äî not to slow you down, but to help you scale with confidence.

Three Growth Stages

Early On: Make It Easy to Contribute

Goal: Encourage participation and quickly identify what's worth standardizing

  • Speed and experimentation matter most
  • Contributions can start small
  • Documentation can be lightweight
  • Components may solve a narrow use case first
  • Iteration after publishing is expected

As You Grow: Make It Easier to Reuse

Goal: Ensure components work reliably across multiple teams and use cases

  • More teams adopt your system
  • Components get reused in more places
  • Changes affect more products
  • Clear guidance becomes more important
  • Documentation becomes more detailed
  • Reviews focus more on reuse and consistency
  • Ownership and maintenance start to matter

At Scale: Make It Safe to Trust

Goal: Maintain trust and reliability so teams can adopt with confidence

  • Teams rely on shared components to move faster
  • "Ready" means safe to use anywhere
  • Contributions are built to last
  • Reviews protect downstream teams
  • Higher bar maintains trust in the system

One Workflow, Flexible Standards

Lunch Box supports the Design System contribution workflow at every stage ‚Äî from early pilots to mature systems.

What changes over time isn't how you contribute, but what "good" looks like:

  • Early systems optimize for learning
  • Growing systems optimize for reuse
  • Mature systems optimize for reliability

You can adopt higher standards gradually, at your own pace.

Our Goal

Contribution guidelines aren't about gatekeeping ‚Äî they're about:

  • Reducing rework
  • Making components easier to adopt
  • Helping your system scale sustainably

Start simple. Improve as you go. Lunch Box is built to support your Design System at every stage.

Contribution Workflows

Four main pathways for contributing patterns and components

Designer Contribution Process

A designer creates a new component in Figma, works through design reviews, and hands off to the engineering team.

1
Designer identifies new component

Component stems from system gaps or new project requirements

2
Create Figma branch

Isolate work from published library using shared libraries

3
Iterate on component design

Ensure alignment with existing components and design tokens

4
Design review with system team

Feedback on consistency, guidelines, and system alignment

5
Merge to library

System team merges branch, adds Figma Variables, notifies designer

6
Trigger General Designer Workflow

Proceed to pattern documentation and design system updates

General Designer Workflow: New Pattern

Create pattern documentation in Lunch Box with Figma design source and prepare for development.

1
Create new pattern in Lunch Box

Establish pathway for integrating new component

2
Link Figma design source & Variables

Ensure alignment between design and code assets

3
Document pattern thoroughly

Include purpose, rationale, UX considerations, and usage guidelines

4
Request design review

Collaborative review with design team for quality and consistency

5
Update status to Ready for Dev

Publish pattern and make available to development team

6
Release (if Figma Variables updated)

Semantic version release and token npm package update

General Engineering Workflow: New Pattern

Implement, test, and release a coded component based on design specifications.

1
Create component template

Set up file structure, parameters, and link to pattern page

2
Use Figma design source & apply Variables

Implement using design tokens for consistency

3
Refine implementation

Ensure design alignment, performance, and coding best practices

4
Design review

Validate visual and functional alignment with design

5
Engineering review via pull request

Verify code quality, performance, and system standards

6
Update pattern status to Ready

Signal readiness for use

7
Merge & release

Merge to main, trigger semantic version release

8
Downstream adoption

Teams update package.json and install latest version

Engineering Contribution Process
