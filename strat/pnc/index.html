Technical Architecture: Repository Integration & Component Registration

Component Data Model, Integration Patterns & Implementation Plan

Component Registration Data Model

Component Metadata Schema

Each registered component must include:

  • Component Identity: Name, version, platform (web/iOS/Android/multiplatform), component type (button, card, form, etc.)
  • Source Reference: Repository URL, file path, git commit hash (read-only link to PNC source code)
  • Documentation: Description, usage guidelines, platform-specific implementation notes, accessibility considerations
  • Token Bindings: Component tokens (color, spacing, typography) with platform-specific values (px, dp, sp, pt)
  • Dependencies: Child components, platform frameworks required (SwiftUI, Kotlin Compose, React, etc.)
  • Status & Governance: Contribution status (proposed, approved, deprecated), owner, review approver, last updated date
  • Multi-Platform Guidance: Platform-specific usage notes (iOS: VoiceOver considerations, Android: TalkBack considerations, Web: keyboard navigation)

Repository Integration Architecture

Bitbucket API Integration Pattern

Component registration workflow:

  • OAuth Credentials: PNC provides Bitbucket OAuth credentials (read-only access scope)
  • Repository Scope: Define which repositories are in scope (e.g., "web-components", "ios-swiftui", "android-kotlin", "design-tokens")
  • Component Discovery: Knapsack queries Bitbucket API to list available components or PNC teams manually register with source links
  • Read-Only Access: Knapsack reads component metadata, documentation, and code references - PNC maintains complete code control in source repos
  • Versioning & Sync: Component versions tracked in Bitbucket; Knapsack syncs metadata on registration update (automatic or manual trigger)
  • Rate Limiting & Performance: API call throttling to respect Bitbucket rate limits; caching for performance

Week 3 (Nov): OAuth Setup & Repository Access Validation

Self-Service Component Registration Workflow

Phase 1b-1c (Dec 2025 - Feb 2026) enablement:

  • Team Access: PNC engineers/designers navigate Knapsack web client, select "Register Component"
  • Metadata Entry: Form-based registration with required fields (component name, repository reference, platform, owner, documentation)
  • Approval Workflow: Component requires design review + engineering validation before publication
  • Automatic Sync: Once approved, component metadata auto-syncs with source repository (Bitbucket) on updates
  • Visibility: Approved components visible to cross-functional teams in web client documentation hub

Key Gaps

  • No Unified Governance: Token approval/versioning process undefined; no audit trail for changes
  • Manual Tracking: Documentation fragmented across Figma, code, and design specs - teams must manually keep aligned
  • Platform Unit Inconsistency: Web (px), iOS (pt), Android (dp/sp) handled separately with no unified view
  • No Versioning: Token changes not tracked; impossible to identify when token values changed or why

Knapsack Token Integration Model (Phase 1a - Dec 2025)

Integration Philosophy: "Integrates With, Not Replaces"

Knapsack does not replace PNC's existing token infrastructure. Instead, Knapsack acts as the centralized source of truth for token documentation, configuration, and governance while integrating with PNC's token sourcing systems.

Platform-Specific Token Unit Support

  • Web (CSS/React): Pixels (px) - absolute units
  • iOS (SwiftUI): Points (pt) - logical units, scales with device resolution
  • Android (Kotlin): Density-Independent Pixels (dp) and Scale-Independent Pixels (sp) - device-agnostic units

Requirement: Knapsack must support conversion/mapping between units so designers can see platform-equivalent values for a given token

Token Sourcing & Sync Strategy (Phase 1a Setup - Nov 2025)

Week 1-2: Token Source Audit
Identify current token sources: Figma token plugin? Coded in repos? Design specs only? Define integration points for each platform
Week 3: Integration Design
Knapsack and PNC architect token data flow: where tokens originate, how they sync to Knapsack, which system is authoritative
Week 4: Token Configuration in Knapsack
Knapsack web client token interface configured with platform-specific units (px, dp, sp, pt) ready for showcase rendering
Week 5: December 2025 Showcase Readiness
Native showcase apps can render components with correct token values; token infrastructure validated

Token Governance Model Definition (Phase 1a - Nov 2025)

Governance Workflow

  • Token Proposal: Team proposes new token or token change in Knapsack with business rationale
  • Review & Approval: Design + Engineering sign-off on token value, platform impact, and compliance
  • Publication: Approved token goes live in Knapsack and syncs to source systems (Figma, code repos)
  • Change Tracking: All token changes logged in audit trail (value, date, approver, reason) - foundation for Q2 2026 versioning enhancement

December 2025 Prerequisite: Token Configuration Operational

For native showcase apps to render correctly, must have:

  • All platform-specific tokens configured in Knapsack (px, dp, sp, pt)
  • Token values synced from source systems (Figma or code)
  • Governance workflow operational (approval process defined)
  • Showcase apps can retrieve token values and render components with correct spacing, colors, typography

Q2 2026 Versioning Roadmap Foundation

Phase 1 setup enables future investment: Q2 2026 "Token Versioning with Audit Trail" enhancement builds on Phase 1 infrastructure. Requires separate investment approval but foundation laid during Phase 1a-b.

Accessibility Standards & Testing Framework

Phase 1 Documentation Approach with Q4 2026 Automation Roadmap

Phase 1: Documentation-Centralization Approach (Nov 2025 - Feb 2026)

Phase 1 Focus: Centralize accessibility standards and provide teams with clear documentation, guidelines, and testing checklists. Automated validation deferred to Q4 2026 investment decision.

Accessibility Documentation Scope

  • WCAG 2.1 AA Standards: Level AA conformance required for all components
  • Web Accessibility: Keyboard navigation, screen reader support (ARIA labels), focus management, color contrast (4.5:1 minimum)
  • iOS Accessibility: VoiceOver support, Dynamic Type scaling, accessible labels and hints, gesture alternatives
  • Android Accessibility: TalkBack support, accessible labels and descriptions, touch target sizing (48x48 dp minimum), content description alternatives
  • Multi-Platform Guidance: Platform-specific implementation notes for designers and engineers

Assessment Required

  • Manual Testing Approach: What's current process for accessibility validation (team-based, automated tools, external audit)?

Q4 2026 Automated Validation Investment Decision Foundation

Phase 1 sets groundwork for Q4 2026 enhancement, but does NOT commit to it. Separate investment approval required.

What Q4 2026 Automated Validation Would Include

  • Automated Tap Target Measurement: iOS (44x44 pt) and Android (48x48 dp) minimum enforcement
  • Gesture Accessibility Checking: Validate alternative interaction paths for complex gestures
  • Color Contrast Validation: Automated checking against WCAG 4.5:1 text and 3:1 graphics standards
  • Text Scaling Testing: Validate responsive behavior at 150%, 200% text size
  • CI/CD Integration: Accessibility validation as automated gate in development pipeline
  • Compliance Reporting: Automated accessibility audit reports for regulatory/compliance documentation

Q4 2026 Decision Gate Inputs

  • Business case for automation: cost/benefit vs. manual testing
  • Regulatory/compliance requirements that drive automation need
  • Testing infrastructure ownership and resource model
  • Integration with PNC's existing testing infrastructure (SauceLabs, CI/CD)

  • Platform-specific accessibility guidelines (iOS, Android, web) clear and comprehensive
  • Component accessibility testing checklist operational
  • Current testing infrastructure baseline established
  • Teams trained on accessibility requirements and testing approach
  • 100% of components meet WCAG 2.1 AA documentation standards by Phase 1c

Objective

Deploy iOS and Android native showcase applications that render PNC's registered SwiftUI/Kotlin components with real-time parameter polling from web client, enabling GNFT/Ledger teams to iterate rapidly on component implementation without requiring app store deployments. Initiative Overview

Architecture Model

Web Client = Source of Truth for documentation, token configuration, and parameters. Native Apps = Component Renderers that poll web client in real-time and display PNC's registered components from Swift UI/Kotlin repositories. PNC maintains complete control of component source code in own repositories.

  • Component registration frameworks for PNC iOS/Android repositories

  • Mobile-specific token rendering (density points, scale-independent pixels)
  • SauceLabs transition strategy and parallel-run or sunset timeline

  • Native showcase usage analytics and adoption tracking
